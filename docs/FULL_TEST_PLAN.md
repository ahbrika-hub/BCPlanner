### admin — POSITIVE
- Dashboard = **Operational**; Business-Lines (weekly) tab visible.
- Tasks: create/edit/approve/assign/close/reject/return/cancel/reopen/**delete**; bulk approve.
- Approvals queue; Recurring/Templates (`recurring.manage`); Users (`users.manage`, `roles.manage`); Projects CRUD (`/admin/projects`); Settings; **Audit** (`audit.read`); Reports + delayed; Performance evaluate; Request-update.
- **NEGATIVE:** none expected (full access).

### section_head — POSITIVE
- Dashboard = **Operational**; weekly tab; **Request update**.
- Approvals (approve/assign/close/reject/return/cancel), bulk approve; Tasks create/edit; Recurring/Templates (`recurring.manage`); Users (`users.manage`); Settings; **Audit**; Reports + delayed; Performance evaluate; view project health `/projects/<id>`.
- **NEGATIVE (must be blocked/hidden):** `/admin/projects` manage (no `projects.manage` → "Access restricted"); task **delete**; the **Executive** dashboard (`dashboard.executive`); `roles.manage` UI.

### employee — POSITIVE
- Dashboard = **Personal** only. Tasks: create, edit own/assigned (per `tasks.update` + RLS), log progress, comment, **submit for review**, upload/download attachments, see project name on a task.
- Reports (own-scope), Workload (read), Performance (own), Notifications bell.
- **NEGATIVE (must be blocked/hidden):**
  - **No `/dashboard/weekly`** (no `dashboard.read` → "Access restricted"); no exec/operational dashboard.
  - **No Approvals**, no approve/assign/close/reject/return/cancel/reopen on tasks.
  - No **Audit**, no **Users**, no **Settings**, no **Recurring/Templates**, no **Projects admin**.
  - No **Request update** button; no `reports.read_all` (delayed report = "Access restricted"); no `performance.read_all`.

### ceo — POSITIVE (READ-ONLY)
- Dashboard = **Executive** (`dashboard.executive`) + **CEO Office Comment** capability (`task_comments.create`).
- **Request update** on the weekly dashboard (`dashboard.request_update`).
- Read **all** tasks (`tasks.read_all`), Reports (`reports.read_all` + delayed), Performance (`performance.read_all`), Workload (`workload.read_all`), view projects, download attachments, Notifications.
- **NEGATIVE (must be blocked/hidden):**
  - **No create/approve/assign/update/close** of tasks (no `tasks.create`/`tasks.update`/approve/assign…); "New Task" hidden, action bar empty.
  - No **Approvals**, no **Audit**, no **Users/Settings/Recurring**, no project **management**.
  - The edit-task dialog must refuse (server-side) even if invoked.

## 2 · Cross-role end-to-end flows
Each: who's logged in · exact path · expected outcome · ☐ pass.

**1) Full lifecycle.** ☐
- *employee1* → Tasks → **New Task** → create (status `pending_approval`). ☐
- *section_head* → Approvals → **Approve** that task → **Assign** to employee1. ☐
- *employee1* → open task → **Log Progress** (update) → **Submit for Review**. ☐
- *section_head* → Approvals (Pending Review) → **Close** (closure summary + rating). Task = `completed`. ☐

**2) Reject / Return / Reopen.** *section_head* on a `pending_approval` task → **Reject** (reason) → status `rejected`; on a `pending_review` → **Return for Modification** (reason); on a terminal task → **Reopen** → `reopened`. (Seed already has examples in each state.) ☐

**3) Bulk approve.** *section_head* → Approvals → tick several **Pending Approval** rows → **Approve** → confirm → per-row results; invalid/out-of-scope rows fail with a reason, valid ones flip. ☐

**4) Weekly dashboard live-on-acceptance.** ☐
- *section_head* (or *ceo*) → `/dashboard/weekly` → **Request update** → a `Dashboard Update` task is created (assigned), notifications go to assignee + section_heads + admins (not ceo); button shows **"Update in progress"** + disabled. ☐
- *employee* (assignee) → open that task → upload the weekly **`.xlsx`** (manual Storage step) on it. ☐
- *section_head* → **Complete** that Dashboard Update task → `/dashboard/weekly` refreshes to the new snapshot (live); a second **Request update** de-dups (no new task). ☐

**5) Notifications.** *employee1* → bell shows the **6 seeded unread** → select-all → **Mark read** / **Mark unread** / **Delete**; counts update. ☐

**6) Reports + date range + CSV + delayed.** *admin/section_head* → Reports → set **From=2026-01-01 To=2026-06-14** → rows span all six months → **Export CSV**; open **Delayed report** (`reports.read_all`) → overdue breakdown by employee/line/priority. *employee* → Delayed report = "Access restricted". ☐

**7) Performance.** *section_head* → Performance → an employee shows the seeded **2025-Q4 / 2026-Q1 / 2026-Q2** history (quarterly periods) and a trend; create/edit an evaluation; sanity-check the 40/30/30 composite. *employee* sees only their **own**. ☐

**8) Recurring.** *admin/section_head* → Recurring → the active **"[TEST] Weekly status report"** → **Generate due now** → spawns a task; the soft-deleted **"[TEST] Monthly archive"** → **Restore**. ☐

**9) Project health.** *admin* → `/admin/projects` → open **"[TEST] Platform Rebuild"** → rollup (totals / completion % / overdue from its linked tasks). *section_head/ceo* reach `/projects/<id>` via a project-type task's Project link. *employee* cannot. ☐

**10) Templates = Recurring.** There is no separate `task_templates` table or
"create from template" selector in the task form — the **Templates** concept is
the **Recurring** feature. *admin/section_head* → Recurring → confirm the two
seeded `[TEST]` rows act as templates (each **Generate due now** spawns a real
`pending_approval` task). (Same surface as flow 8.) ☐

**11) CEO.** *ceo* → Dashboard = **Executive**; open any task → add a **CEO Office Comment** (visible on the timeline). Confirm "New Task"/action bar are absent. ☐

**12) Overdue.** *any manager* → Tasks → **Overdue** filter / badge shows the **12 seeded overdue** tasks (due `< 2026-06-14`, still active). ☐

**13) Signup decision.** Logged out → `/signup` → `someone@saptco.com.sa` → **accepted** (decision/redirect; email won't arrive without SMTP); `someone@gmail.com`, `x@saptco.com`, `x@hotmail.com` → **rejected** inline. ☐

## 3 · Teardown (MARKER-ONLY · USER-PRESERVING)
Run `supabase/sample-data/full-test-teardown.sql` (SQL editor). It deletes **only**
rows carrying a seed marker (`[TEST]` titles/names, `evaluation_notes` `[TEST]…`,
`raw_file_path='test-seed'`, and children scoped via their `[TEST]` parent task),
in FK-safe child→parent order, in one transaction (idempotent).

**No user is ever removed** — it NEVER deletes from `public.profiles` or
`auth.users`, and NEVER scopes a delete by a user id. A real user's own
(non-marked) tasks, notifications, evaluations and comments are left completely
untouched. `audit_logs` and reference tables (business_lines, permissions,
role_permissions, app_settings, departments) are also untouched.

The final SELECTs must show **Verification A** = every marked count **0**, and
**Verification B** = `real_users_found = 6`, `auth_users_found = 6`, and the
reference tables non-zero.

**Reminder:** attachments uploaded to Storage and any `audit_logs` from your test
actions are *not* covered by the SQL teardown — remove Storage objects manually
if desired (audit logs are immutable history and can be left).
