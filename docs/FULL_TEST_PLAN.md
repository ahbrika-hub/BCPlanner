### section_head ‚Äî POSITIVE
- Dashboard = **Operational**; weekly tab; **Request update**.

- **NEGATIVE (must be blocked/hidden):** `/admin/projects` manage (no `projects.manage` ‚Üí "Access restricted"); task **delete**; the **Executive** dashboard (`dashboard.executive`); `roles.manage` UI.

### employee ‚Äî POSITIVE
- Dashboard = **Personal** only. Tasks: create, edit own/assigned (per `tasks.update` + RLS), log progress, comment, **submit for review**, upload/download attachments, use a **template** in the create form, see project name on a task.

- **NEGATIVE (must be blocked/hidden):**
  - **No `/dashboard/weekly`** (no `dashboard.read` ‚Üí "Access restricted"); no exec/operational dashboard.
  - **No Approvals**, no approve/assign/close/reject/return/cancel/reopen on tasks.
  - No **Audit**, no **Users**, no **Settings**, no **Recurring**, no **Projects/Templates admin**.
  - No **Request update** button; no `reports.read_all` (delayed report = "Access restricted"); no `performance.read_all`.

### ceo ‚Äî POSITIVE (READ-ONLY)
- Dashboard = **Executive** (`dashboard.executive`) + **CEO Office Comment** capability (`task_comments.create`).
- **Request update** on the weekly dashboard (`dashboard.request_update`).
- Read **all** tasks (`tasks.read_all`), Reports (`reports.read_all` + delayed), Performance (`performance.read_all`), Workload (`workload.read_all`), view projects/templates, download attachments, Notifications.
- **NEGATIVE (must be blocked/hidden):**
  - **No create/approve/assign/update/close** of tasks (no `tasks.create`/`tasks.update`/approve/assign‚Ä¶); "New Task" hidden, action bar empty.
  - No **Approvals**, no **Audit**, no **Users/Settings/Recurring**, no project/template **management**.
  - The edit-task dialog must refuse (server-side) even if invoked.

## 2 ¬∑ Cross-role end-to-end flows
Each: who's logged in ¬∑ exact path ¬∑ expected outcome ¬∑ ‚òê pass.

**1) Full lifecycle.** ‚òê
- *employee1* ‚Üí Tasks ‚Üí **New Task** ‚Üí create (status `pending_approval`). ‚òê
- *section_head* ‚Üí Approvals ‚Üí **Approve** that task ‚Üí **Assign** to employee1. ‚òê
- *employee1* ‚Üí open task ‚Üí **Log Progress** (update) ‚Üí **Submit for Review**. ‚òê
- *section_head* ‚Üí Approvals (Pending Review) ‚Üí **Close** (closure summary + rating). Task = `completed`. ‚òê

**2) Reject / Return / Reopen.** *section_head* on a `pending_approval` task ‚Üí **Reject** (reason) ‚Üí status `rejected`; on a `pending_review` ‚Üí **Return for Modification** (reason); on a terminal task ‚Üí **Reopen** ‚Üí `reopened`. (Seed already has examples in each state.) ‚òê

**3) Bulk approve.** *section_head* ‚Üí Approvals ‚Üí tick several **Pending Approval** rows ‚Üí **Approve** ‚Üí confirm ‚Üí per-row results; invalid/out-of-scope rows fail with a reason, valid ones flip. ‚òê

**4) Weekly dashboard live-on-acceptance.** ‚òê
- *section_head* (or *ceo*) ‚Üí `/dashboard/weekly` ‚Üí **Request update** ‚Üí a `Dashboard Update` task is created (assigned), notifications go to assignee + section_heads + admins (not ceo); button shows **"Update in progress"** + disabled. ‚òê
- *employee* (assignee) ‚Üí open that task ‚Üí upload the weekly **`.xlsx`** (manual Storage step) on it. ‚òê
- *section_head* ‚Üí **Complete** that Dashboard Update task ‚Üí `/dashboard/weekly` refreshes to the new snapshot (live); a second **Request update** de-dups (no new task). ‚òê

**5) Notifications.** *employee1* ‚Üí bell shows the **6 seeded unread** ‚Üí select-all ‚Üí **Mark read** / **Mark unread** / **Delete**; counts update. ‚òê

**6) Reports + date range + CSV + delayed.** *admin/section_head* ‚Üí Reports ‚Üí set **From=2026-01-01 To=2026-06-14** ‚Üí rows span all six months ‚Üí **Export CSV**; open **Delayed report** (`reports.read_all`) ‚Üí overdue breakdown by employee/line/priority. *employee* ‚Üí Delayed report = "Access restricted". ‚òê

**7) Performance.** *section_head* ‚Üí Performance ‚Üí an employee shows the seeded **2025-Q4 / 2026-Q1 / 2026-Q2** history (quarterly periods) and a trend; create/edit an evaluation; sanity-check the 40/30/30 composite. *employee* sees only their **own**. ‚òê

**8) Recurring.** *admin/section_head* ‚Üí Recurring ‚Üí the active **"[TEST] Weekly status report"** ‚Üí **Generate due now** ‚Üí spawns a task; the soft-deleted **"[TEST] Monthly archive"** ‚Üí **Restore**. ‚òê

**9) Project health.** *admin* ‚Üí `/admin/projects` ‚Üí open **"[TEST] Platform Rebuild"** ‚Üí rollup (totals / completion % / overdue from its linked tasks). *section_head/ceo* reach `/projects/<id>` via a project-type task's Project link. *employee* cannot. ‚òê

**10) Create from template.** *admin/section_head* ‚Üí `/admin/templates` ‚Üí confirm the two seeded **"[TEST] Weekly Status Report"** / **"[TEST] Incident Review"** templates (create/edit one to exercise `templates.manage`). Then *employee1* ‚Üí Tasks ‚Üí **New Task** ‚Üí **template selector** ‚Üí pick **"[TEST] Weekly Status Report"** ‚Üí fields pre-fill (title/description/priority/business line) ‚Üí submit ‚Üí normal `pending_approval` task. ‚òê

**11) CEO.** *ceo* ‚Üí Dashboard = **Executive**; open any task ‚Üí add a **CEO Office Comment** (visible on the timeline). Confirm "New Task"/action bar are absent. ‚òê

**12) Overdue.** *any manager* ‚Üí Tasks ‚Üí **Overdue** filter / badge shows the **12 seeded overdue** tasks (due `< 2026-06-14`, still active). ‚òê

**13) Signup decision.** Logged out ‚Üí `/signup` ‚Üí `someone@saptco.com.sa` ‚Üí **accepted** (decision/redirect; email won't arrive without SMTP); `someone@gmail.com`, `x@saptco.com`, `x@hotmail.com` ‚Üí **rejected** inline. ‚òê

## 3 ¬∑ Teardown (MARKER-ONLY ¬∑ USER-PRESERVING)
Run `supabase/sample-data/full-test-teardown.sql` (SQL editor). It deletes **only**
rows carrying a seed marker (`[TEST]` titles/names, `evaluation_notes` `[TEST]‚Ä¶`,
`raw_file_path='test-seed'`, and children scoped via their `[TEST]` parent task),
in FK-safe child‚Üíparent order, in one transaction (idempotent).

**No user is ever removed** ‚Äî it NEVER deletes from `public.profiles` or
`auth.users`, and NEVER scopes a delete by a user id. A real user's own
(non-marked) tasks, notifications, evaluations and comments are left completely
untouched. `audit_logs` and reference tables (business_lines, permissions,
role_permissions, app_settings, departments) are also untouched.

The final SELECTs must show **Verification A** = every marked count **0**, and
**Verification B** = `real_users_found = 6`, `auth_users_found = 6`, and the
reference tables non-zero.

**Reminder:** attachments uploaded to Storage and any `audit_logs` from your test
actions are *not* covered by the SQL teardown ‚Äî remove Storage objects manually
if desired (audit logs are immutable history and can be left).
